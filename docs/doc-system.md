# Documentation System

## 目标

这个项目已经不是一个“单页静态站改改样式”级别的工作量了，后续会同时推进：

- 移动端产品与交互持续迭代
- `Word -> 豆包 -> JSON` 的内容生产后端
- 景点评价的真实评论与互动后端
- 登录、存储、审核、发布与运维

因此文档必须从“说明型文档集合”升级成“治理型文档系统”。

## 当前问题

当前 `docs/` 有不少有效内容，但仍存在 4 个结构缺口：

1. 缺总盘。
   没有一份文档统一说明“现在到底在推进哪些任务线、优先级是什么、当前进度到哪了”。
2. 缺现行方案层。
   `phases/` 更像历史过程文档，不适合承担未来架构推进的主入口。
3. 缺决策记录层。
   重要架构判断目前更多散落在聊天和临时说明里，没有沉淀成可追溯的 ADR。
4. 缺更新规则。
   文档之间有主次之分，但没有被明确写成规则。

## 文档分层

后续 `docs/` 按下面 5 层维护：

### 1. Index

- `README.md`
  文档总入口，只负责“从哪里开始看”和“当前哪些文档有效”。

### 2. Current State

这层回答“项目现在是什么样”：

- `architecture.md`
- `content-pipeline.md`
- `source-materials.md`
- `regression-checklist.md`

### 3. Active Planning

这层回答“接下来要做什么、按什么方案做”：

- `roadmap.md`
- `backend-foundation.md`
- 后续新增的专题方案文档

### 4. Decision Records

这层回答“为什么这么定”：

- `adr/adr-xxx-*.md`

ADR 只记录高影响、会影响后续实现方向的判断，不记录一般性 UI 调整。

### 5. History

这层回答“这个项目是怎么演进到今天的”：

- `phases/*.md`

`phases/` 保留，但默认不作为新任务的主入口。

## 状态定义

路线图与专项方案统一使用下面 5 个状态：

- `proposed`
  已提出，但还没开始进入实现排期。
- `active`
  已明确进入当前推进队列。
- `blocked`
  方向明确，但受依赖限制暂时不能推进。
- `done`
  当前阶段目标已完成。
- `archived`
  不再推进，或已经被新方案替代。

## 更新规则

1. 改当前运行时结构：
   必须同步 `architecture.md`。
2. 改内容生产链路：
   必须同步 `content-pipeline.md` 和对应专项方案。
3. 改高影响技术方向：
   必须新增或更新一条 ADR。
4. 开新任务线：
   先更新 `roadmap.md`，再开专项文档。
5. 老 `phase` 文档不追求逐条补齐：
   如果和现行方案冲突，以现行方案和 ADR 为准。

## 推荐工作流

1. 先看 `docs/README.md`
2. 再看 `docs/roadmap.md`
3. 再看对应专项方案文档
4. 开工时确认是否需要补 ADR
5. 完成后更新当前状态文档和路线图状态

## 当前最重要的新增任务线

目前最需要进入现行治理层的是两条：

- 内容生产后端化
  目标是把 `Word -> JSON` 从本地脚本升级成正式服务。
- 社区互动后端化
  目标是让评论、点赞、UP、踩从本地存储升级成真实全站数据。
