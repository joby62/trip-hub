# Roadmap

## 目标

这份文档是长期总盘。

它不负责描述实现细节，而是统一回答 3 个问题：

1. 项目长期有哪些任务线
2. 每条任务线的目标和依赖是什么
3. 下一阶段整体优先顺序是什么

如果你要看“当前状态、最近更新、是否结项”，应该去看 `current-status.md`。

## 状态说明

- `active`：当前推进中
- `proposed`：方案已明确，待排期
- `blocked`：依赖未满足
- `done`：当前阶段目标完成
- `archived`：不再继续

## 任务线拆解

### T1 移动端产品收口

- 目标
  继续把当前页面做成移动端可读、可刷、可点的稳定成品。
- 当前状态
  已有基础页面和交互，但仍处于快速调整期。
- 交付边界
  只处理前端呈现，不承担真实后端能力。

### T2 内容生产后端化

- 目标
  支持用户上传 Word，由后端生成结构化 JSON，再由前端直接读取渲染。
- 当前状态
  已完成 docs 级方案收敛，当前仍是本地脚本流水线，下一步进入服务化实现准备。
- 当前依赖
  需要先明确 JSON schema、任务状态模型、对象存储和豆包调用方式。
- 第一阶段交付
  `上传 -> 解析 -> 豆包结构化 -> 校验 -> 产出 JSON`

### T3 社区互动后端化

- 目标
  景点评价页的评论、点赞、UP、踩都必须是多人共享的真实数据。
- 当前状态
  已完成 docs 级方案收敛，当前仍是浏览器本地存储，只对当前设备有效。
- 当前依赖
  需要登录策略、数据库表设计、评论图片存储和审核边界。
- 第一阶段交付
  先做真实评论与真实计数，不先追求复杂社区能力。

### T4 账号与风控

- 目标
  让互动和内容导入具备最小身份可信度。
- 当前状态
  已完成登录、鉴权和审核边界文档设计，待实现。
- 第一阶段交付
  读匿名、写登录；操作日志可追溯；评论可审核。

### T5 发布与运维

- 目标
  支撑 JSON 发布、对象存储、异步任务和 API 服务。
- 当前状态
  当前运行形态仍是轻后端静态壳，已完成任务状态机梳理，待落基础设施。
- 第一阶段交付
  先把单机服务升级成“API + DB + Redis + TOS”的基础部署形态。

## 下一阶段优先顺序

### P0

- 建最小 FastAPI API 骨架
- 落本地 PostgreSQL / Redis / worker / TOS 开发形态
- 固化 `render_json` schema contract 和 `point_key` 规则

### P1

- 建最小后端骨架
- 打通 Word 上传与异步任务
- 打通评论与反应的真实读写接口

### P2

- 接入登录
- 补评论图片上传
- 补审核与限流

## 和状态板的分工

- `roadmap.md`
  看长期优先级和任务线存在性。
- `current-status.md`
  看当前正在推什么、谁已结项、哪里阻塞。
- `timeline.md`
  看关键节点的时间流。

## 维护要求

1. 新开任务线前，先更新这里。
2. 某条任务线方向被取消时，把状态改成 `archived` 或移除并在 `timeline.md` 说明。
3. 某条任务线的日常推进和结项，不在这里细写，而是更新 `current-status.md`。
