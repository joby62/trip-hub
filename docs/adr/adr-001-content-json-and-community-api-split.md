# ADR 001: Content JSON Delivery And Community API Split

## Status

accepted

## Context

项目未来会同时承载两类能力：

1. 攻略内容生产与发布
2. 景点评价与互动

这两类能力的数据特征完全不同：

- 内容更适合版本化、审核后发布、静态读取、缓存分发
- 社区互动更适合实时写入、实时读出、权限控制和计数聚合

如果把两者混在一个数据层和发布模型里，后续会很难维护。

## Decision

项目采用双线后端架构：

1. 内容走 `render_json` 发布链路
2. 社区互动走实时 API + PostgreSQL 持久化链路

具体表现为：

- 前端主页面内容统一读取已发布 JSON
- 评论、点赞、UP、踩单独通过 API 获取和提交
- JSON 发布和互动写入互不阻塞

## Consequences

### Positive

- 内容发布可缓存、可回滚
- 社区互动可做真实计数与权限控制
- 前端边界清晰，职责稳定

### Negative

- 后端系统会比当前静态壳复杂
- 需要同时维护发布链路和实时链路
- 需要更早引入数据库、对象存储和任务队列

## Notes

后续如果要新增“用户自建攻略”能力，也应优先落在内容生产线，而不是直接塞进社区互动模型。
