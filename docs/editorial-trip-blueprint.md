# Editorial Trip Blueprint

## 目标

在不删除现有功能、不删减任何文档内容、不丢失任何图片的前提下，把当前云南攻略页重组为一套更接近 `stitch 2` 参考稿的 editorial trip 网站。

这次改版的重点不是“再做一个更好看的单页”，而是：

- 保留当前所有功能位和工具位。
- 参考 `jade_horizon/DESIGN.md` 的视觉语言和页面编排。
- 把文档内容和图片按时间逻辑、计划逻辑、景点逻辑重新组织。
- 让页面从“功能栈式攻略页”升级成“可浏览、可搜索、可操作的旅行站”。

## 不可动约束

以下内容是硬约束，后续实现时不能破坏：

1. 不删功能
   当前已有的搜索、避坑筛选、预订工具卡、打包清单、本地持久化、图片灯箱、原文引用、URL 深链必须保留。

2. 不删内容
   `static/guide/source/yunnan_trip_v4/document.md` 和 `document.txt` 中的正文不能裁掉，只能分层组织与重排展示。

3. 不丢图片
   `static/guide/source/yunnan_trip_v4/images/` 中全部图片都必须被组织，并且不能出现孤立图片。

4. 只重做组织方式
   本轮可以重做布局、阅读顺序、视觉层级和交互承载方式，但不能把页面能力降回“纯展示页”。

## 参考设计输入

### 设计系统来源

- `stitch 2/jade_horizon/DESIGN.md`

- Tonal layering 优先于硬边框
- 大图、玻璃、留白、非对称边距
- 强调高端旅行手册感，而不是工具面板感

- `stitch 2/daily_itinerary/code.html`
- `stitch 2/attraction_detail/code.html`

这些原型不是逐像素复刻模板，而是页面职责模板。布局和视觉安排可以高度参考，功能逻辑仍以当前工程为准。

## 当前工程与参考稿的差距

### 当前页面的优点

- `static/guide/yunnan.js` 已具备比较完整的攻略功能：
  - 全屏搜索
  - 避坑筛选
  - 预订工具卡
  - 打包清单
  - 图片灯箱
  - 原文模式
  - `day / image / tool` 深链

- 文档和图片已经落地到本地素材层：
  - `static/guide/source/yunnan_trip_v4/document.md`
  - `static/guide/source/yunnan_trip_v4/manifest.json`
  - `static/guide/source/yunnan_trip_v4/day-map.json`
  - `static/guide/source/yunnan_trip_v4/images/`

### 当前页面的主要问题

1. 页面是“功能区块堆栈”，不是“旅行站信息架构”
2. Hero 仍偏说明页，不够像目的地入口
3. 每天内容主要通过详情层承载，适合工具浏览，不够适合完整图文阅读
4. 图片已经接入，但“按时间”和“按景点”的双重组织还不够明确
5. 原文已经存在，但还没有成为主叙事的一部分

## 新的总信息架构

本次改版建议保留 `/guides/yunnan` 作为主入口，但在内部升级成 4 个 screen-level 主视图：

1. `总览`
2. `行程`
3. `景点`
4. `清单`

这 4 个主视图对应 stitch 2 的 4 个参考页面职责，同时继续复用当前站点的全局搜索、筛选、深链、图片和原文能力。

## 四大主视图蓝图

### 1. 总览视图

参考：`trip_overview/code.html`

职责：

- 用大图 Hero 建立目的地气质
- 先让用户理解路线主骨架
- 暴露关键决策入口
- 暴露工具型信息，但不让工具区破坏叙事

结构建议：

1. 顶部玻璃导航
   - Logo / 标题
   - 搜索按钮
   - 可保留分享或更多入口

2. 大图 Hero
   - 顶部 35% - 45% 为主视觉
   - 覆盖标题、时间范围、路线状态
   - 用渐变叠层承托标题，不再以纯文字块为主

3. Highlight Bento
   - 3 到 4 个 trip highlights
   - 例如：湖景、公路、高原、雪山

4. Trip Facts
   - 天数
   - 路线跨度
   - 高原起点
   - 重点预订项

5. 路线主骨架
   - 昆明 → 大理 → 泸沽湖 → 香格里拉 → 梅里 → 丽江
   - 视觉上参考 `trip_overview` 的章节式时间轴

6. 关键行动卡
   - 当前 `bookingTools`
   - 当前 `pitfallFilters`
   - 当前 `overviewTools`
   - 改成 editorial 卡片层，而不是工具面板平铺

总览视图原则：

- 它不是信息终点，而是旅程入口
- 先激发浏览欲，再进入行程

### 2. 行程视图

参考：`daily_itinerary/code.html`

职责：

- 承载完整的 Day 1 - Day 11 结构
- 让用户按时间读完整个旅程
- 保留当前的日程功能，但用更强的章节化方式承载

结构建议：

1. 顶部日期 rail
   - `Day 1` 到 `Day 11`
   - 横向滚动
   - 当前日高亮

2. 单日章节视图
   每天不再只是摘要卡片 + 抽屉，而是一个可完整进入的日程章节：
   - 当天主图
   - 日期 / 城市 / 海拔 / 节奏
   - 当天一句判断
   - 交通块
   - 景点块
   - 餐饮块
   - 住宿块
   - 避坑块
   - 原文入口

3. 日内编排
   文档原文未必严格按“上午/中午/下午/晚上”写作，因此需要支持两种组织：
   - 第一层按“出发 / 到达 / 景点 / 餐饮 / 住宿 / 注意事项”
   - 第二层保留“完整原文顺序”

4. 图片位置
   - 图片不能只在图库 tab 里
   - 图片应进入行程主叙事，在对应段落或对应计划块附近出现

5. 原文模式
   - 继续保留
   - 不再是附属功能，而是单日页的正式一层

行程视图原则：

- 每一天都要像一本旅行手册中的单独章节
- 工具信息是辅助，不是主体阅读节奏

### 3. 景点视图

参考：`attraction_detail/code.html`

职责：

- 解决“用户按景点决策”的浏览需求
- 把文档中反复出现的核心目的地整理成独立页面级内容
- 把图片和原文做第二次聚合，而不是只按天散落

建议的核心景点实体：

- 昆明滇池
- 海东 / 理想邦
- 双廊
- 喜洲
- 沙溪
- 泸沽湖
- 独克宗古城
- 松赞林寺
- 纳帕海
- 飞来寺 / 梅里
- 虎跳峡
- 玉龙雪山
- 蓝月谷
- 白沙古镇

每个景点视图应包含：

- 景点 Hero
- 一句话价值判断
- 推荐去法 / 费用 / 最佳时间
- 关联图片
- 关联原文段落
- 关联哪几天
- 避坑提醒
- 预订建议

景点视图原则：

- 它是“按计划逻辑组织图片和文本”的核心入口
- 同一张图片可以同时属于某一天和某个景点

### 4. 清单视图

参考：`travel_checklist/code.html`

职责：

- 承接当前所有工具型能力
- 避免工具模块破坏总览与行程的主叙事

清单视图内部建议拆成 3 个分区：

1. 行前清单
   - 当前 `packingList`
   - 当前本地持久化逻辑
   - 当前分组折叠逻辑

2. 重要备忘
   - 当前 `globalNotes`
   - 当前预订提醒
   - 当前高原提醒

3. 关键预订
   - 当前 `bookingTools`
   - 当前 `bookingList`

清单视图原则：

- 功能完全保留
- 视觉风格转为更统一的 tonal cards，而不是独立工具条块拼接

## 内容与图片组织规则

### 规则 1：所有图片必须双归属

每张图至少必须有两个维度：

- `day_id`
- `attraction_id` 或 `theme_id`

如果某张图不适合归入具体景点，也必须归入某个主题组，例如：

- 公路转场
- 住宿空间
- 湖景视角
- 高原自然

### 规则 2：所有原文必须完整保留

原文不能删，只能分层：

- `策展摘要`
- `章节组织内容`
- `完整原文模式`

其中“完整原文模式”必须始终可达。

### 规则 3：所有内容至少有两种浏览逻辑

1. 时间逻辑
   - Day 1 - Day 11

2. 计划逻辑
   - 景点
   - 交通
   - 餐饮
   - 住宿
   - 避坑
   - 预订

### 规则 4：禁止图片孤立存在

每张图片都必须能回答 3 个问题：

- 这张图属于哪一天？
- 这张图属于哪一个景点或计划节点？
- 这张图对应哪段原文？

## 现有功能保留映射

| 当前功能 | 改版后承载位置 | 保留要求 |
| --- | --- | --- |
| 全屏搜索 | 全站顶部搜索层 | 保留全屏、分组搜索、结果直达 |
| 避坑筛选 | 总览视图 + 行程视图 + 景点视图 | 保留分类，不降级为静态说明 |
| 预订工具卡 | 总览视图 + 清单视图 | 保留可点跳转和过滤动作 |
| 打包清单 | 清单视图 | 保留本地存储、折叠、批量操作 |
| 图片灯箱 | 行程视图 + 景点视图 | 保留上一张/下一张/回原文 |
| 原文模式 | 单日章节 + 景点页引用 | 保留完整原文可达性 |
| 深链 | 全站 | 从 `day/image/tool` 扩展为 `view/day/image/tool/attraction` |

## 数据层蓝图

建议在现有素材层上方增加一层统一数据视图，例如：

`static/guide/data/yunnan.blueprint.json`

建议结构：

```json
{
  "trip": {},
  "days": [],
  "attractions": [],
  "media": [],
  "tools": [],
  "notes": []
}
```

### days

每一天需要包含：

- 基础摘要
- 当天主图
- 当天图片列表
- 关联原文段落
- 计划块
- 关联景点
- 避坑标签
- 预订标签

### attractions

每个景点需要包含：

- 标题
- 英文别名或副标题
- 归属天数
- 归属图片
- 归属原文段落
- 费用 / 预约 / 交通 / 摄影建议

### media

每张图片建议包含：

- `sequence`
- `src`
- `day_id`
- `attraction_ids`
- `source_block_ids`
- `caption`
- `context_before`
- `context_after`

## 视觉系统落地原则

### 要继承的参考稿规则

1. Hero 大图优先
2. Glass Top Bar / Bottom Nav
3. 无硬分隔线，尽量用 tonal shifts
4. 大圆角、低阴影、层次靠色面
5. `Plus Jakarta Sans` 用于大标题 / 数字 / metadata 强调
6. 中文主体保持高可读、较高行高

### 当前页面需要被替换的视觉倾向

当前 `static/guide/yunnan.css` 更偏：

- 暖纸张感
- 功能卡片堆叠
- 较明显阴影和边框感

后续需要转向：

- teal / ochre / moss 的 horizon palette
- 明显更“蓝绿地平线”
- section 之间用色阶，不用边框切断
- 卡片更像画册和 guidebook spread

## 交互承载原则

### 保留底部导航，但改职责

底部导航建议改为：

- `总览`
- `行程`
- `景点`
- `清单`

不再使用“图片”作为一级视图，因为图片应该进入行程和景点两个主叙事层。

### 搜索仍为全局入口

搜索层继续保留，但结果目标扩大为：

- 某一天
- 某景点
- 某张图
- 某段原文
- 某工具模块

### 详情层不再是唯一主承载

当前 detail sheet 可以保留，但定位调整为：

- 快速预览层
- 或小屏二级承载层

真正的大块内容应进入 screen-level 主视图。

## 实施顺序

### Stage A：信息架构重组

- 在 `yunnan.html` 中引入 4 个主视图容器
- 先不删现有功能逻辑
- 把当前模块映射到新视图

### Stage B：数据组织升级

- 建立 `day + attraction + media + source` 四层映射
- 把全部图片双归属化
- 把原文段落增加章节归类

### Stage C：页面布局重构

- 重做总览视图
- 重做行程视图
- 新增景点视图
- 重做清单视图

### Stage D：视觉系统替换

- 统一色板
- 统一字体层级
- 替换边框/阴影策略
- 重做 top bar / bottom nav / hero / card 语言

### Stage E：交互整合

- 搜索目标升级
- 深链升级
- 图片在多视图内复用
- 原文和图片双向跳转稳定化

## 验收标准

1. 页面视觉上明显向 stitch 2 的 editorial 方向靠拢，而不是当前功能卡片堆栈。
2. 当前已有功能一个都不能消失。
3. 文档正文可以完整访问，不能只剩摘要。
4. 全部图片都能在“按天”和“按景点/计划”至少一处找到。
5. 用户既可以按 Day 浏览，也可以按景点决策。
6. 移动端仍然是主设计对象，桌面端只做拉开和增强。

## 本蓝图与旧方案文档的关系

  - editorial 信息架构
