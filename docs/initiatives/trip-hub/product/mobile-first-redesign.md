# Mobile-First Redesign

## 目标

把当前“桌面感较强的长页攻略”改成一套移动端优先的旅行网页：

- 手机上先成立，再向桌面增强。
- 首先解决单手操作、信息决策和图片浏览。
- 让提取出来的图片与原文真正进入页面主叙事，而不是继续停留在摘要信息页。

## 当前页面的主要问题

1. 当前结构偏 desktop
   左侧 sticky 导航、宽屏摘要区、并排模块都更像桌面看板。

2. 当前内容偏摘要
   11 天卡片是二次总结，不是完整攻略。图片、原文、细节语气还没有接进页面。

3. 当前交互偏整页重渲染
   搜索、筛选、展开都通过 `renderAll()` 完成，移动端会显得笨重。

4. 当前素材层与页面层分离
   `static/guide/source/yunnan_trip_v4` 已经有图文素材，但页面没有消费它们。

5. 当前工具区还不完整
   页面里有避坑区容器，但 JavaScript 没真正渲染；移动端也没有便捷入口。

## 移动端目标信息架构

### 第一屏

第一屏只保留最关键的决策信息：

- 路线标题
- 时间范围
- 总路线简图
- 三个主按钮：`看总览`、`按天看`、`先看避坑/抢票`

不在第一屏塞入大量摘要卡和并排信息块。

### 底部固定操作栏

移动端改为底部导航，不再依赖左侧栏：

- `总览`
- `行程`
- `图片`
- `清单`

搜索做成单独入口按钮，点击后进入全屏搜索层。

### 行程浏览

移动端的主列表应该是“按天摘要卡片流”：

- 日期
- 城市
- 一句话判断
- 1 张封面图
- 3 个标签，例如 `拍照`、`长车程`、`高原`

点卡片进入当天详情，而不是在列表里展开成很长的一大块。

### 当天详情

当天详情建议做成全屏页或底部抽屉页，内部再分四个标签：

- `路线`
- `图片`
- `吃住`
- `原文`

这样用户在手机上不需要反复穿过长列表。

### 工具区

把下面这些内容改成可单独进入的工具型模块：

- 避坑标签
- 抢票提醒
- 高原提示
- 打包清单

## 核心交互原则

### 日程卡片

- 列表页只展示摘要，不展示整块详情。
- 点击卡片进入详情页。
- 详情页内切换标签，不整页重绘整个列表。

### 图片浏览

- 每一天详情顶部直接有图片图廊。
- 图廊支持横向滑动。
- 点击图片进入全屏灯箱。
- 灯箱里要显示图片对应的原文段落或上下文摘要。

### 搜索

搜索结果不能只搜摘要，至少覆盖：

- 天数标题
- 景点名
- 餐馆名
- 原文段落
- 图片引用上下文

### 避坑筛选

把避坑做成可点击筛选标签，而不是静态文案：

- `收费不值`
- `建议绕开`
- `必须提前订`
- `高原提醒`

### 打包清单

- 移动端改成单列大按钮勾选。
- 继续保留本地存储。
- 支持一键展开 / 收起分类。

## 数据层整改

后续页面要同时使用三层数据：

1. `dayData`
   保留为摘要层，负责日程卡片上的一句话信息。

2. `static/guide/source/yunnan_trip_v4/manifest.json`
   负责图片和原文上下文引用。

3. `static/guide/source/yunnan_trip_v4/document.md` / `document.txt`
   负责原文层和搜索层。

最终应整理出一个中间数据层，把“某一天对应哪些图片、哪些原文段落、哪些避坑标签”明确映射出来。

## 技术实现原则

- CSS 以移动端宽度为默认样式，再向上做平板和桌面增强。
- 列表、详情、图库、搜索分别管理状态，不再依赖整页 `renderAll()`。
- 优先保证点击路径短、可单手触达、滚动层级简单。
- 桌面端只做增强排版，不再单独主导布局。

## 验收标准

- 手机首屏一屏内能完成路线认知和主入口选择。
- 任意一天都能在 2 次点击内进入详情。
- 图片可以独立浏览，并能找到对应原文。
- 搜索结果能回到具体天数或具体图片。
- 打包清单在手机上可以顺手勾选，不依赖桌面布局。

## 开发顺序

- 先做 `phases/phase1.md`
- 再做 `phases/phase2.md`
- 最后做 `phases/phase3.md`
